## 1. Agent ID System
### Overview
Every agent (client) in Alpha Hub has a unique `agent_id` that serves as the **universal routing key** for all lead delivery, Google Ads tracking, landing page URLs, and CRM subaccount mapping. **Alpha Hub is the source of truth for agent IDs вҖ” not GHL.**
### Generation
Agent IDs are generated by the `generateAgentId()` function in `supabase/functions/agent-onboarding-webhook/index.ts`:
```typescript
function generateAgentId(): string {
  const chars = 'abcdefghijklmnopqrstuvwxyz0123456789';
  let result = '';
  for (let i = 0; i < 20; i++) {
    result += chars.charAt(Math.floor(Math.random() * chars.length));
  }
  return result;
}
```
**Properties:**
- **Length**: 20 characters
- **Character set**: Lowercase alphanumeric only (`a-z`, `0-9`)
- **Generated once** at client creation time, never changes
- **Stored in**: `clients.agent_id` column
### Legacy Mixed-Case IDs
Some older agents have mixed-case agent IDs (e.g., `AbCdEfGh12345678xy`). These are preserved in the database for backward compatibility because:
- Existing Webflow landing page URLs use the original casing
- Google Ads campaign `finalUrlSuffix` parameters reference the original ID
- External links (thank-you pages, NFIA pages, scheduler pages) may embed the ID
**Rule**: New agent IDs are always lowercase. Legacy mixed-case IDs are never modified. If a legacy agent has URL-related issues, all profile links (scheduler, thank-you, lander) must be manually synchronized.
### Important ID Distinctions
There are **three** different GHL-related IDs on the `clients` table, and they serve different purposes:
| Column | What It Stores | Purpose |
|--------|---------------|---------|
| `agent_id` | Alpha Hub-generated 20-char string | **Primary routing key** for leads, ads, URLs. Source of truth. |
| `ghl_contact_id` | GHL Contact ID from the **sales CRM** | Used for SaaS billing activation. This is the prospect's contact record in the main Alpha Agent CRM (location `wDoj91sbkfxZnMbow2G5`). |
| `ghl_agent_ref` | GHL Contact ID sent in the webhook payload's `agent_id` field | Preserved for debugging only. This was historically used before Alpha Hub generated its own IDs. |
| `ghl_user_id` | GHL User ID in the agent's **own subaccount** | Used for lead assignment (`assignedTo`) when injecting leads. Must be a valid user in that specific subaccount. |
| `subaccount_id` | GHL Location ID of the agent's subaccount | The target location for lead injection. |

## 2. Client Onboarding Flow
### Overview
When a new agent signs up (books an onboarding call), the GHL Main CRM fires a webhook to `agent-onboarding-webhook`. This edge function creates the client record and triggers a 19-step automated onboarding process.
### Edge Function: `agent-onboarding-webhook`
**File**: `supabase/functions/agent-onboarding-webhook/index.ts` (1,093 lines)
**Auth**: API key via `x-api-key` header, validated against `webhook_api_keys` table
### Step-by-Step Flow
#### Phase 1: Authentication & Validation
1. **API Key Validation**: The `x-api-key` header is checked against the `webhook_api_keys` table. The key must exist and be `is_active = true`. On each valid request, `last_used_at` and `request_count` are updated.
2. **Payload Parsing**: The webhook payload from GHL is parsed. It supports multiple field naming conventions:
   - `first_name`, `firstName`, `First_Name`, `First Name`
   - `email`, `Email`, `contact_email`
   - `phone`, `Phone`, `contact_phone`
   - `states`, `States`, `licensed_states`, `stateslicensed`
   - etc.
3. **Contact ID Extraction**: Three separate identifiers are extracted:
   - `contactId` (root-level `contact_id`) вҖ” for SaaS billing
   - `ghlContactId` (from `agent_id` field in payload) вҖ” legacy agent reference
   - `prospectId` вҖ” to link to existing prospect record
4. **Required Field Validation**: `email` and `name` are required. Returns 400 if missing.
#### Phase 2: Deduplication Check
5. **Existing Client Check**: Query `clients` by email. If a match is found вҶ’ **rebooking flow** (updates existing record, re-triggers automation). If no match вҶ’ **new client flow**.
#### Phase 3A: Rebooking Flow (Existing Client)
6. Update the client record with new `onboarding_call_scheduled_at`, phone, contact IDs
7. Skip states update if client has an active Google Ads campaign (to prevent targeting changes)
8. Deduplication: check if an `onboarding_automation_runs` record was created in the last 60 seconds вҖ” if so, bail out
9. Create a new `onboarding_automation_runs` record (status: `pending`)
10. Trigger `run-full-onboarding` edge function (idempotent вҖ” skips already-completed steps)
#### Phase 3B: New Client Flow
11. **Generate Agent ID**: Call `generateAgentId()` to create a 20-character lowercase alphanumeric ID
12. **Create Auth User**: `supabase.auth.admin.createUser()` with email + temp password + `email_confirm: true`. If user already exists, find and reuse the existing auth user.
13. **Insert Client Record**: Insert into `clients` table with:
    - `agent_id` = generated ID
    - `ghl_contact_id` = sales CRM contact ID
    - `ghl_agent_ref` = legacy agent reference
    - `status` = `'onboarding'`
    - `onboarding_status` = `'in_progress'`
    - `automation_started_at` = now
14. **Create Profile & Role**: Upsert `profiles` record, insert `user_roles` with `role = 'client'`
15. **Create Onboarding Tasks**: Insert 16 default onboarding task records into `onboarding_tasks` table
16. **Enroll in Courses**: Query published courses and bulk-insert `enrollments` records
17. **Stabilize Headshot**: If `profile_image_url` is provided, call `refresh-stable-headshot` to download and re-host the image to Supabase Storage (`media` bucket). This ensures the headshot URL is reliable for Webflow pages and calendar pages that embed it.
18. **Link Prospect**: Try to find matching prospect by `prospect_id` or by email:
    - Set `prospects.client_id = client.id` and `prospects.converted_at = now`
    - Set `clients.prospect_id = prospect.id` (bidirectional link)
    - Pull `ad_spend_budget` and `referral_code` from prospect if not already set
19. **Process Referral**: If a `referral_code` is present:
    - Look up `referral_codes` table
    - Prevent self-referral
    - Set `clients.referred_by_client_id`
    - Insert or update `referrals` record with `status = 'active'`
20. **Create Billing Records**:
    - Management fee вҶ’ `billing_records` (type: `management`, recurrence: `monthly`)
    - Ad spend вҶ’ `billing_records` (type: `ad_spend`, recurrence from payload or `bi_weekly`)
    - Client wallet вҶ’ `client_wallets` (balance: 0, threshold: 150)
21. **Trigger Automation**: After a 3-second delay (to ensure DB writes are committed), call `run-full-onboarding` edge function
22. **Fire Sale Conversion**: Call `google-ads-enhanced-conversion` with `conversionType: 'Agent_Sale_API'` ($1,497 value)
### The 19-Step Automation (`run-full-onboarding`)
**File**: `supabase/functions/run-full-onboarding/index.ts` (2,037 lines)
Each step is tracked in `onboarding_automation_runs`:
- `steps_completed`: array of completed step numbers
- `steps_failed`: array of failed step numbers
- `step_data`: JSON object with output from each step
- `error_log`: array of error entries with timestamps
- `current_step`: currently executing step
**Concurrency Protection**: Before starting, the function checks for existing `running` runs and uses atomic `UPDATE ... WHERE status != 'running'` to prevent duplicate execution.
| Step | Name | What It Does |
|------|------|-------------|
| 1 | `lowercase_name` | Normalize agent name to proper case |
| 2 | `generate_slug` | Generate URL slug from agent name (e.g., `john-smith`) |
| 3 | `generate_bio` | Call `generate-agent-bio` to create AI-written agent bio |
| 4 | `create_nfia` | Create NFIA (National Financial Insurance Alliance) page via Webflow CMS API |
| 5 | `create_scheduler` | Create scheduler page via `webflow-cms-create` |
| 6 | `create_lander` | Create landing page via `webflow-cms-create` |
| 7 | `create_profile` | Create TFWP profile page via `webflow-cms-create` |
| 8 | `create_thankyou` | Create thank-you page via `webflow-cms-create` |
| 9 | `create_subaccount` | Create GHL subaccount via `ghl-create-subaccount`, then create GHL user via `ghl-create-user` |
| 10 | `activate_saas` | **Manual step** вҖ” SaaS activation requires human intervention |
| 11 | `install_snapshot` | Verify CRM snapshot status, poll until calendar is available |
| 12 | `pull_calendar_id` | Discover calendar IDs via `crm-discovery-calendar` |
| 13 | `assign_calendars` | Assign GHL user to all calendars via `ghl-assign-user-to-all-calendars` |
| 14 | `update_scheduler_embed` | Update Webflow scheduler page with calendar embed code |
| 15 | `sync_crm_custom_fields` | Sync custom field mappings via `ghl-sync-custom-fields` |
| 16 | `create_google_ads` | Create Google Ads campaign via `create-google-ads-campaign` |
| 17 | `final_verification` | Run verification checks on all artifacts |
| 18 | `verify_onboarding` | Final test: send test lead, verify delivery |
| 19 | `provision_phone` | **Manual step** вҖ” Twilio phone provisioning |
**Idempotency**: Each step checks if the artifact already exists (e.g., `client.subaccount_id` is non-empty вҶ’ skip step 9). When resuming from a later step, `hydrateStepsFromClientArtifacts()` credits prior steps based on existing data.

### Edge Function: `lead-webhook`
**File**: `supabase/functions/lead-webhook/index.ts` (471 lines)
**Auth**: API key via `x-api-key` header
#### Step-by-Step:
1. **API Key Validation**: Same pattern as onboarding webhook вҖ” checks `webhook_api_keys` table.
2. **Field Normalization**: Supports ALL CAPS (GHL native), camelCase, snake_case, and space-separated field names:
   ```
   agent_id | agentId | Agent_ID | Agent ID | AGENT_ID | AGENT ID
   lead_id  | leadId  | Lead_ID  | Lead ID  | LEAD_ID  | LEAD ID
   email    | Email
   phone    | Phone
   state    | State   | Resident State
   first_name | firstName | First_Name | First Name
   ```
3. **Lead ID Generation**: Priority order:
   - `contact_id` from GHL (best for idempotency)
   - `lead_id` from payload
   - `crypto.randomUUID()` as fallback
4. **Date Parsing**: Supports multiple formats:
   - ISO strings
   - MM/DD/YYYY (US format)
   - Nested `contact.dateAdded`, `contact.dateCreated`
   - Falls back to `new Date()` if all parsing fails
5. **Idempotency Check**: Query `leads` table by `lead_id`. If duplicate found, return `200` with `duplicate: true` вҖ” **no re-insertion**.
6. **Pipeline Metric**: `increment_pipeline_metric(agent_id, 'webhook_received')`
7. **Insert Lead**: Store in `leads` table with all parsed fields + `webhook_payload` (raw JSON for debugging)
8. **Pipeline Metric**: `increment_pipeline_metric(agent_id, 'stored')`
9. **Identity Resolution** (see [Attribution System](#5-attribution-system) for details):
   - Link `visitor_id` to lead via `visitor_sessions`
   - Build `lead_attribution` record with first/last touch data
   - Calculate `time_to_conversion_hours`
10. **Set Delivery Status**: Update lead to `delivery_status: 'pending'`
11. **Fire-and-Forget Injection**: Call `inject-lead-to-ghl` asynchronously:
    ```typescript
    fetch(`${supabaseUrl}/functions/v1/inject-lead-to-ghl`, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${supabaseServiceKey}`,
      },
      body: JSON.stringify({ leadId: newLead.id }),
    }).then(res => { ... }).catch(err => { ... });
    ```
    This is non-blocking вҖ” the webhook returns immediately to the caller.
12. **Return Response**: `201` with `lead_id` and `success: true`
### Edge Function: `inject-lead-to-ghl`
**File**: `supabase/functions/inject-lead-to-ghl/index.ts` (629 lines)
**Auth**: Service role key via `Authorization` header (internal calls only)
#### Step-by-Step:
1. **Fetch Lead**: Load full lead record from `leads` table by `leadId`
2. **Client Lookup**: Query `clients` by `agent_id`:
   ```typescript
   const { data: client } = await supabase
     .from('clients')
     .select('id, subaccount_id, ghl_user_id, name, crm_delivery_enabled')
     .eq('agent_id', lead.agent_id)
     .single();
   ```
   **Fallback**: If no match by `agent_id`, the system also supports lookup by `url_slug`.
3. **CRM Delivery Check**: If `client.crm_delivery_enabled === false`, mark lead as `skipped` and return
4. **Subaccount Validation**: If `client.subaccount_id` is null, mark lead as `failed`
5. **Get Company ID**: Fetch from `ghl_oauth_tokens` table (single agency record)
6. **Get Location Token**: Call `crm-location-token` to exchange agency token for location-scoped token
7. **Load Field Mappings**: Query `ghl_custom_field_mappings` for this client to get ID-based custom field mappings
8. **Build GHL Contact Payload** with all lead data, custom fields, tags, and assignment
9. **Custom Field Strategy**:
   - **If ID-based mappings exist**: Use `{ id: "ghl_field_id", value: "..." }` format вҖ” guaranteed to work
   - **If no mappings**: Fall back to key-based: `{ key: "contact.savings", value: "..." }` вҖ” may not work
   - **Fallback field**: Always includes a `fallback` custom field with ALL lead data formatted as readable text
10. **Create Contact**: POST to `https://services.leadconnectorhq.com/contacts/`
11. **Duplicate Handling**: If GHL returns 400 with "duplicate":
    - Extract `contactId` from error response `meta.contactId`
    - OR search by email/phone
    - Add `alpha` tag to existing contact
    - Update custom fields on existing contact
12. **Log Delivery**: Insert into `lead_delivery_logs` and update lead's `delivery_status`
13. **Pipeline Metrics**: `increment_pipeline_metric(agent_id, 'delivered')` or `'failed'`
### Delivery Status Lifecycle
```
pending вҶ’ delivered    (success)
pending вҶ’ failed       (first failure)
failed  вҶ’ retrying     (retry picked up)
retrying вҶ’ delivered   (retry success)
retrying вҶ’ failed      (retry failure, attempts < 3)
failed  вҶ’ failed_permanent (attempts >= 3)
```
### Edge Function: `retry-failed-lead-delivery`
**File**: `supabase/functions/retry-failed-lead-delivery/index.ts` (124 lines)
**Trigger**: Cron job or manual invocation
- Fetches up to **50 leads** with `delivery_status IN ('failed', 'pending')` and `delivery_attempts < 3`
- For each lead: sets status to `retrying`, calls `inject-lead-to-ghl`
- If attempts reach `MAX_RETRY_ATTEMPTS` (3): sets `delivery_status = 'failed_permanent'`
### Pipeline Metrics
The `lead_pipeline_metrics` table tracks counts at each stage per agent per day:
| Stage | When Incremented |
|-------|-----------------|
| `webhook_received` | Lead webhook receives valid payload |
| `stored` | Lead successfully inserted into DB |
| `delivered` | GHL contact created/updated successfully |
| `failed` | GHL injection fails |
| `skipped` | CRM delivery disabled for agent |

## 4. GHL OAuth & Token Bridge
### Overview
Alpha Hub uses a **single agency-level OAuth connection** to GoHighLevel (GHL) that serves all agent subaccounts. This is the "master key" pattern вҖ” one connection, many locations.
### Architecture

### Token Encryption (AES-256-GCM)
All tokens are encrypted before storage:
- **Storage Format**: Base64-encoded string containing `[12-byte IV][encrypted data][16-byte GCM auth tag]`
- **Encryption Key**: Stored in `ENCRYPTION_KEY` secret. Padded/truncated to exactly 32 bytes.
- **Critical Note**: External systems (e.g., Python AI agents) that need to decrypt tokens must use the exact same encryption key and AES-256-GCM parameters.
### How One Connection Serves All Agents
```
Agency OAuth Token (encrypted in ghl_oauth_tokens)
    в”Ӯ
    в”ңв”Җв”Җ Agent A: crm-location-token(companyId, locationId_A) вҶ’ Token A
    в”ңв”Җв”Җ Agent B: crm-location-token(companyId, locationId_B) вҶ’ Token B
    в””в”Җв”Җ Agent C: crm-location-token(companyId, locationId_C) вҶ’ Token C
```

## 5. Attribution System
### Overview
Alpha Hub tracks visitor behavior across external landing pages using a client-side JavaScript tracking script for first-touch / last-touch attribution, Google Ads enhanced conversion reporting, referral code tracking, and conversion path analysis.
### Client-Side: Tracking Script
**File**: `supabase/functions/tracking-script/index.ts` (287 lines)
Embedded on external pages via: `<script src="https://{supabase-url}/functions/v1/tracking-script"></script>`
Key behaviors:
- **Visitor ID**: Persistent via localStorage (`v_` prefix)
- **Session ID**: Session-scoped via sessionStorage (`s_` prefix), 30-minute timeout
- **Smart Source Detection**: UTM params вҶ’ Click IDs (gclid/fbclid) вҶ’ Referrer parsing вҶ’ Direct fallback
- **First/Last Touch**: First touch stored once, last touch updated every page load
- **Auto-Tracked Events**: `session_start`, `page_view`, `click` (data-track elements), `form_submit`
### Identity Resolution
When a lead arrives with a `visitor_id`:
1. Find all visitor sessions for that visitor
2. Lock sessions by setting `email`, `lead_id`, `converted_at`
3. Build `lead_attribution` record with first/last touch, conversion path, time-to-conversion

## 6. Referral System
### Overview
Uses a **first-touch wins** attribution model. Referral codes are auto-generated on client creation from the agent's name.
### Flow
1. Prospect visits `/apply?ref=JohnSmith`
2. `prospect-contact-capture` stores referral code, triggers `link_prospect_to_referrer`
3. On conversion to client, referral status changes from `pending` вҶ’ `active`
4. Self-referral prevention built into both triggers and onboarding webhook

## 7. Prospect /apply Flow
### Overview
Multi-step Typeform-style qualification form that captures prospect info and syncs to GHL sales CRM.
### 2-Step GHL Sync Strategy
- **Step 1** (`prospect-contact-capture`): Native fields only (name, email, phone) + `qual-pending` tag
- **Step 2** (`prospect-qualification-submit`): Custom fields (savings, investments, etc.) + `form-complete` tag
- All prospects sync to hardcoded location: `wDoj91sbkfxZnMbow2G5`

## 9. Key Database Tables Reference
### `clients`

### `leads`
All captured leads: `lead_id` (idempotency key), `agent_id`, `delivery_status`, `delivery_attempts`, `ghl_contact_id`, UTM fields, `webhook_payload` (raw JSON)
### `ghl_oauth_tokens`
Single-row agency OAuth: encrypted `access_token`, `refresh_token`, `expires_at`, `company_id`
### `ghl_custom_field_mappings`
Per-client field mappings: `client_id`, `field_name`, `ghl_field_id`, `ghl_field_name`
### `visitor_sessions`
Tracking sessions: `visitor_id`, `session_id`, UTMs, `gclid`, `fbclid`, `referrer_url`, `landing_page`, `email`, `lead_id`, `converted_at`
### `lead_attribution`
Full attribution per lead: first/last touch fields, `touch_count`, `total_page_views`, `total_sessions`, `time_to_conversion_hours`, `conversion_path`
### `prospects`
Qualification form data: `visitor_id`, `email`, `name`, `phone`, `referral_code`, `referrer_client_id`, `qualified_path`, `ghl_contact_id`, `partial_answers`, `client_id`
### `onboarding_automation_runs`
19-step automation progress: `client_id`, `status`, `current_step`, `steps_completed`, `steps_failed`, `step_data`, `error_log`
### `lead_pipeline_metrics`
