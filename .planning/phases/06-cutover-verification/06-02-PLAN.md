The MCP proxy endpoint:
```
POST https://qcunascacayiiuufjtaq.supabase.co/functions/v1/mcp-proxy
Headers: Content-Type: application/json, x-mcp-secret: ${MCP_PROXY_SECRET}
```

**Run these verification calls:**

```bash
#!/bin/bash
SUPABASE_URL="https://qcunascacayiiuufjtaq.supabase.co"
MCP_SECRET="${MCP_PROXY_SECRET}"
PASS=0
FAIL=0
RESULTS=""

verify_tool() {
  local tool="$1"
  local params="${2:-{}}"
  local response

  response=$(curl -s -w "\n%{http_code}" -X POST \
    "${SUPABASE_URL}/functions/v1/mcp-proxy" \
    -H "Content-Type: application/json" \
    -H "x-mcp-secret: ${MCP_SECRET}" \
    -d "{\"tool\": \"${tool}\", \"params\": ${params}}")

  local http_code=$(echo "$response" | tail -1)
  local body=$(echo "$response" | sed '$d')

  if [ "$http_code" = "200" ] && echo "$body" | jq -e '.result' > /dev/null 2>&1; then
    echo "PASS: ${tool} (HTTP ${http_code})"
    PASS=$((PASS + 1))
    RESULTS="${RESULTS}PASS: ${tool}\n"
  else
    echo "FAIL: ${tool} (HTTP ${http_code})"
    echo "  Response: $(echo "$body" | head -c 200)"
    FAIL=$((FAIL + 1))
    RESULTS="${RESULTS}FAIL: ${tool} (HTTP ${http_code})\n"
  fi
}

echo "=== Feature Verification ==="
echo ""

# Feature 1: Client Management (CUT-05)
echo "--- Client Management ---"
verify_tool "list_clients" '{"limit": 5}'
verify_tool "search_clients" '{"query": "warren"}'
verify_tool "get_client_detail" '{"client_name": "James Warren"}'

# Feature 2: Billing (CUT-06)
echo "--- Billing ---"
verify_tool "get_billing_summary" '{}'
verify_tool "get_stripe_invoices" '{"limit": 5}'
verify_tool "get_stripe_charges" '{"limit": 5}'
verify_tool "get_stripe_subscriptions" '{"limit": 5}'

# Feature 3: Ad Spend Wallets (CUT-07)
echo "--- Ad Spend Wallets ---"
verify_tool "get_ad_spend_overview" '{}'

# Feature 4: Campaign Health (CUT-08)
echo "--- Campaign Health ---"
verify_tool "get_campaign_health" '{}'

# Feature 5: Lead Pipeline (CUT-09)
echo "--- Lead Pipeline ---"
verify_tool "get_lead_pipeline" '{}'

# Feature 6: Communications (CUT-10)
echo "--- Communications ---"
verify_tool "get_communications" '{"limit": 5}'

# Feature 7: Financial Projections & Alerts (CUT-11)
echo "--- Financial Projections & Alerts ---"
verify_tool "get_financial_projections" '{}'
verify_tool "get_alerts" '{}'

echo ""
echo "=== RESULTS ==="
echo "Passed: ${PASS}"
echo "Failed: ${FAIL}"
echo ""
if [ "$FAIL" -gt 0 ]; then
  echo "VERDICT: FAIL -- ${FAIL} tool(s) did not pass"
  echo "Fix failures before proceeding with webhook switchover"
else
  echo "VERDICT: PASS -- All ${PASS} tools verified"
fi
```

**Pass criteria:** ALL tools must return HTTP 200 with a `.result` field in the JSON response.

**Triage guidance when tools fail:**

If a tool returns an error, diagnose using this decision tree:

1. **HTTP 401/403 (auth error):** Check that `MCP_PROXY_SECRET` env var is set and matches the secret on the new project. Verify via: `supabase secrets list --project-ref qcunascacayiiuufjtaq | grep MCP_PROXY_SECRET`

2. **HTTP 500 with "relation does not exist" or "column does not exist" (schema error):** The re-sync may have missed a table or column. Check: `SELECT tablename FROM pg_tables WHERE schemaname = 'public' ORDER BY tablename;` and compare with old project.

3. **HTTP 200 but `.result` is empty or null (data error):** Data was not imported for this table. Verify the specific table has rows: `SELECT count(*) FROM <table>;`. If empty, re-run the bridge function for that specific table.

4. **HTTP 200 but `.result` contains an error message (function error):** Read the edge function logs: check Supabase Dashboard > Edge Functions > mcp-proxy > Logs. Common causes: missing secrets (check `supabase secrets list`), incorrect env var names, or a code bug.

5. **HTTP 504 (timeout):** The mcp-proxy query is too slow. May indicate missing indexes after re-sync. Run `ANALYZE;` on the new DB to refresh statistics, then retry.

6. **Stripe-related tools fail (get_stripe_invoices, etc.):** These call Stripe API directly -- they do not depend on re-synced data. Check that `STRIPE_SECRET_KEY` and `STRIPE_AD_SPEND_SECRET_KEY` secrets are set on the new project.

**Distinguish error types before fixing:** Data errors (from re-sync) need bridge re-run. Function errors need code/secret fixes. Config errors need secret updates. Do NOT re-run the full re-sync for a single table failure -- just re-run the bridge for that specific table.

**This is a go/no-go gate.** Do NOT proceed to the checkpoint until all tools pass.
