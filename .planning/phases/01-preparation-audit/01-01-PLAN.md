<task type="auto">
  <name>Task 2: Audit codebase and create CODEBASE.md</name>
  <files>.planning/inventories/CODEBASE.md</files>
  <action>
Create `.planning/inventories/` directory and produce `CODEBASE.md` with the following sections:

**1. Repository Overview**
- Latest commit hash, date, total file count
- Top-level directory structure with purpose of each

**2. Edge Functions Inventory (all ~92)**
For each function directory in `supabase/functions/`:
- Name
- Purpose (derived from function name and reading the index.ts)
- Key dependencies (other functions it imports, external APIs it calls)
- JWT verification setting from `supabase/config.toml` (verify_jwt = true/false)

Categorize functions into groups: billing, CRM/GHL, Google Ads, lead management, communications, auth, Stripe webhooks, MCP, cron/scheduled, utilities, other.

**3. Database Tables Inventory (all ~111)**
Parse the migration files in `supabase/migrations/` to extract CREATE TABLE statements. For each table:
- Name
- Category (billing, clients, leads, campaigns, communications, auth-related, system)
- Key relationships (foreign keys)
- Whether it has RLS enabled (look for `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`)

**4. Frontend Structure**
- Key directories under `src/`
- Page components and their routes
- Supabase client configuration location
- Stripe.js integration points (look for `VITE_STRIPE_*` usage)

**5. Critical Investigation: Stripe Key Locations**
Research identified ZERO `Deno.env.get()` calls for Stripe secrets in edge functions. Investigate:
- Search for `sk_live`, `sk_test`, `whsec_`, `stripe.com/v1`, `Stripe(` across entire codebase
- Check if Stripe keys are stored in database tables
- Check frontend for `VITE_STRIPE_*` env var usage
- Document WHERE Stripe API keys are actually stored and HOW Stripe server-side calls are made
- This finding directly affects Phase 4 (Stripe Migration) planning

**6. LOVABLE_API_KEY Investigation**
- Read the `generate-agent-bio` edge function to determine what `LOVABLE_API_KEY` is used for
- Document whether this key is needed post-migration or if a replacement is needed

**7. Suspicious Items / Flags**
- Dead code or unused functions
- Deprecated patterns
- Potential issues for migration

Format: Use markdown tables for the function and table inventories. Keep descriptions concise (one line each).
