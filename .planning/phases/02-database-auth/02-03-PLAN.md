**A. Programmatic Login Test**

Test auth by making a direct API call to the new project's GoTrue endpoint. Pick any user from auth.users (preferably the user's own account -- query auth.users for email addresses first):

```bash
# List user emails to find a test candidate
psql --dbname "$NEW_DB_URL" -c "SELECT email FROM auth.users ORDER BY email LIMIT 10;"
```

Then attempt login via the GoTrue REST API:
```bash
curl -s -X POST \
  "$SUPABASE_V2_URL/auth/v1/token?grant_type=password" \
  -H "apikey: $SUPABASE_V2_ANON_KEY" \
  -H "Content-Type: application/json" \
  -d '{"email": "[USER_EMAIL]", "password": "test"}' \
  | jq '.error // .access_token[:20]'
```

NOTE: We do NOT know any user's actual password, so this API call will likely return "Invalid login credentials" -- which is EXPECTED and actually GOOD. It proves:
1. The GoTrue auth endpoint is working on the new project
2. The auth.users table is accessible
3. The authentication flow is operational

What would be BAD is: "User not found" or a 500 error -- that would indicate the auth migration failed.

If the response is "Invalid login credentials" (HTTP 400), that confirms auth is working but we just do not know the password. The user will verify with their real password in the checkpoint task.

**B. RLS Functional Test via API**

Test that the anon key is properly blocked from accessing protected data:

```bash
# Should return empty array or error (NOT actual client data)
curl -s \
  "$SUPABASE_V2_URL/rest/v1/clients?select=id,company_name&limit=5" \
  -H "apikey: $SUPABASE_V2_ANON_KEY" \
  -H "Authorization: Bearer $SUPABASE_V2_ANON_KEY" \
  | jq .
```

```bash
# Should return empty array (NOT actual billing data)
curl -s \
  "$SUPABASE_V2_URL/rest/v1/billing_records?select=id,amount&limit=5" \
  -H "apikey: $SUPABASE_V2_ANON_KEY" \
  -H "Authorization: Bearer $SUPABASE_V2_ANON_KEY" \
  | jq .
```

```bash
# Should return empty array (NOT actual wallet data)
curl -s \
  "$SUPABASE_V2_URL/rest/v1/wallet_transactions?select=id,amount&limit=5" \
  -H "apikey: $SUPABASE_V2_ANON_KEY" \
  -H "Authorization: Bearer $SUPABASE_V2_ANON_KEY" \
  | jq .
```

Each should return `[]` (empty array) or a permission error. If ANY returns actual data rows, RLS is NOT working -- that is a critical security failure.

Test with service role key to confirm data exists (proving RLS is the blocker, not missing data):
```bash
curl -s \
  "$SUPABASE_V2_URL/rest/v1/clients?select=id,company_name&limit=3" \
  -H "apikey: $SUPABASE_V2_ANON_KEY" \
  -H "Authorization: Bearer $SUPABASE_V2_SERVICE_ROLE_KEY" \
  | jq .
```
This SHOULD return actual client data (service role bypasses RLS).
