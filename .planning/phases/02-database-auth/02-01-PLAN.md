```bash

```
If present, these may cause non-fatal errors during restore. The `--single-transaction` flag means any error aborts. If OWNER TO lines cause failures, you will need to either:
- Comment them out with sed: `sed -i '' 's/ALTER .* OWNER TO/-- &/' schema.sql`
- Or try restore first and only apply the fix if it fails

Restore schema (roles + schema in single transaction):
```bash
psql \
  --single-transaction \
  --variable ON_ERROR_STOP=1 \
  --file /Users/forren/workspace/copy-alphahub/.planning/dumps/roles.sql \
  --file /Users/forren/workspace/copy-alphahub/.planning/dumps/schema.sql \
  --dbname "$NEW_DB_URL"
```

If the restore fails due to OWNER TO errors, apply the sed fix and retry. If it fails for other reasons, read the error output carefully -- common issues:
- "role does not exist": Comment out the specific GRANT/ALTER for that role
- "extension already exists": Safe to ignore (use `CREATE EXTENSION IF NOT EXISTS`)
- "relation already exists": The target DB may have Supabase-default tables -- may need `DROP SCHEMA public CASCADE; CREATE SCHEMA public;` before restore (DANGEROUS -- only if needed)
